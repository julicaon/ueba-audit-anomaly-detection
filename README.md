# ueba-audit-anomaly-detection

**Выявление аномальной активности пользователей в журнале аудита (UEBA)**

**Краткое описание:** обучение без учителя для обнаружения учётных записей с отклоняющимся поведением в журнале аудита корпоративной ИС (инсайдерские угрозы, обход регламентов, признаки компрометации).

**Автор(ы):** ФИО, группа  
**Курс:** Технологии искусственного интеллекта и продвинутой аналитики  
**Дата:** Модуль 3–4, 2025/2026

---

## Бизнес-задача

**Проблема:** журнал аудита содержит десятки тысяч событий; ручной разбор не масштабируется и субъективен. Службе внутреннего аудита и информационной безопасности нужен приоритизированный список учётных записей, где активность выходит за рамки типичных бизнес-процессов: массовые операции, использование emergency-режима без обоснования, активность в нерабочее время, критические действия (DELETE, DEACTIVATE) без комментария.

**ML-формулировка:** обнаружение аномалий (unsupervised anomaly detection) на агрегированных поведенческих профилях пользователей.

**Целевая метрика:** согласованность детекции экспертного индекса риска (CRS) и алгоритма Isolation Forest, полнота охвата при гибридном подходе (снижение доли пропущенных кейсов на 30–40% по сравнению с одним методом), ручная верификация топ-22 пользователей. Классические F1-score и ROC-AUC не применимы — размеченных инцидентов в датасете нет.

## Feature Engineering

| Действие | Применённый метод | Обоснование |
|----------|-------------------|-------------|
| Обработка пропусков | Пустая строка для `comment` | Отсутствие комментария — сигнал риска для критических операций |
| Кодирование категорий | Бинарные флаги `is_critical`, `has_comment` | DELETE/DEACTIVATE и наличие обоснования |
| Масштабирование | Min-Max (CRS), StandardScaler (Isolation Forest) | Сопоставимость метрик |
| Новые признаки | `hour`, `day_of_week`, `is_night`, `is_business_hours`, `comment_length`; агрегаты по пользователю: `total_actions`, `emergency_ratio`, `night_ratio`, `no_comment_ratio`, `critical_ratio` | Поведенческий профиль UEBA |

**Разделение выборки:** классический train/validation/test не используется (100 агрегированных профилей). Валидация — сопоставление IQR, CRS и Isolation Forest, ручной разбор топ-22.

**Стратификация:** нет (отсутствуют метки классов).

## Моделирование

> Ноутбук: [`notebooks/02_modeling_crs_isolation_forest.ipynb`](notebooks/02_modeling_crs_isolation_forest.ipynb)

### Baseline

| Модель | Параметры | Результат |
|--------|-----------|-----------|
| IQR по `total_actions` | Q1 = 305, Q3 = 332,5, IQR = 27,5 | 15 пользователей с активностью > 374 |
| Пороговые правила | emergency > 30%, ночь > 25%, критические без комментария > 50% | 17 / 52 / 52 пользователя |

### Эксперименты

| Модель | Параметры (кратко) | Метрика (оценка) | Интерпретация |
|--------|-------------------|------------------|---------------|
| **CRS** | Веса: `norm_emergency` 0,25; `norm_critical`, `norm_no_comment`, `norm_actions` — по 0,20; `norm_night` 0,15; порог высокого риска ≥ 0,45 | 22 пользователя | «Громкие» нарушения регламента |
| **Isolation Forest** | `n_estimators=100`, `contamination='auto'`, `random_state=42` | 22 аномалии | «Тихие» многомерные паттерны |
| **Гибрид IQR + CRS + IF** | Пересечение трёх методов | **9 пользователей** (ядро: ID 5, 18, 21, 60, 70 и др.) | Критический уровень аудита |
| **Строгие пороги (IQR)** | > 374 действий + emergency/ночь > 25% + критические без комментария > 50% | **12 пользователей** | Максимально подозрительное поведение |

**Лучший подход:** гибрид CRS + Isolation Forest с приоритизацией в 3 уровня (критический / повышенный / мониторинг).

**Время обучения:** Isolation Forest на 100 строках — менее 1 с.

## Интерпретация

### Feature Importance (топ-5 по весам CRS)

1. `norm_emergency` (0,25) — обход стандартных проверок  
2. `norm_critical` (0,20) — доля критических операций  
3. `norm_no_comment` (0,20) — действия без обоснования  
4. `norm_actions` (0,20) — интенсивность активности  
5. `norm_night` (0,15) — внеурочная активность  

### Анализ ошибок

- Только CRS, ML = норма — возможна легитимная высокая нагрузка (DevOps, релизы); рекомендуется «белый список» после проверки.  
- Только Isolation Forest — редкие комбинации признаков (например, ID 97, 88, 54, 67).  
- Веса CRS заданы экспертно; нет размеченных исторических инцидентов.

### Вывод для бизнеса

Наиболее рискованны сочетания: объём > 374 действий, emergency и ночная активность > 25%, критические операции без комментария > 50%. Рекомендации: обязательное обоснование для DELETE/DEACTIVATE, 2FA для emergency-режима, алерты в SIEM/SOAR.

Подробности — в [`docs/report.docx`](docs/report.docx).

## MLOps / Инференс

**Сохранение модели**

| Параметр | Значение |
|----------|----------|
| Формат | `joblib` |
| Файл модели | `models/isolation_forest_audit.joblib` |
| Вместе с моделью | `StandardScaler`, список признаков, веса CRS |

**Загрузка и прогноз**

```python
import joblib

bundle = joblib.load("models/isolation_forest_audit.joblib")
model = bundle["model"]
scaler = bundle["scaler"]
features = bundle["features"]

X = scaler.transform(user_stats[features])
user_stats["ml_prediction"] = model.predict(X)
user_stats["ml_anomaly_score"] = model.decision_function(X)
# Расчёт risk_score (CRS) — см. notebooks/02_modeling_crs_isolation_forest.ipynb
```
